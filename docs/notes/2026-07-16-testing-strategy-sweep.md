## Gap list

1. **Same physical ICM mounted into two workspaces** (Registry/workspace) — no test mounts the identical physical ICM path into two separate workspace configs and confirms both stay independently healthy with separate session/audit history (Acceptance Scenario 5's core mechanic).
2. **Unrelated (mounted-but-undeclared) ICM excluded from a launched session's context/read_roots** (Context/launch) — the "declared-but-unmounted" case is tested; a third, actually-mounted-but-undeclared ICM is never introduced in `session_scope_test.exs`/`session_read_roots_test.exs` to prove exclusion, even though the identical fixture pattern exists for link-rewrite/search/backlinks.
3. **Related ICM's own instruction content is not injected/loaded** (Context/launch) — only the pointer (root + entrypoint path) in `context.md` is asserted; no test proves the related ICM's actual CONTEXT.md body is absent from what reaches the harness.
4. **Required-workflow preflight failure for a missing related ICM** (Context/launch) — `SessionScope`'s own moduledoc flags this as "a later phase's concern"; only a task-input-locator-unmounted case is tested, not a workflow declaring a required related ICM.
5. **Frontend live-session dot rendering** — only the underlying boolean/ordering logic is tested; no render-level coverage exists (no Svelte test harness in the repo).
6. **`+` new-session button threads the correct mountKey** (Frontend) — `startSession` lives inline in `IcmProjects.svelte`, not extracted/tested; only the backend RPC given a mount_key is covered.
7. **Knowledge tree renders only in the Knowledge list pane** (Frontend) — placement/exclusivity across panes is untested; only route-param resolution logic is.
