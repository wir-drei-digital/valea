# Valea rigidity map — where the system imposes structure on an ICM or on how work flows

Scope: worktree `.claude/worktrees/workspace-profiles` @ `d099c5b` (branch `feat/workspace-profiles`). Read-only
survey, file:line cited. Companion to `.superpowers/sdd/reference-icm-study.md` (structural grammar of two real,
organically-grown ICMs used as counter-evidence) — that doc answers "what do real ICMs look like"; this one
answers "exactly where does Valea's own code assume they don't look like that."

Redesign hypothesis under test: real ICMs are arbitrary trees of markdown + subfolders + scripts (workflows at
any depth, lowercase, freeform), and Valea should offer email/calendar as filesystem building blocks (e.g.
maildir) the agent connects, rather than fixed pipelines.

---

## 1. Workflow model

**Registry scan — top-level `Workflows/*.md` only, non-recursive, capital-W.**
`Valea.Workflows.workflows_for_mount/1` globs exactly `<mount.root>/Workflows/*.md`
(`backend/lib/valea/workflows.ex:224-231`) — one `Path.wildcard/1` on a fixed, single-segment glob. Nothing
recurses into subfolders. `list/1` (`workflows.ex:74-80`) unions this per-mount glob across every enabled mount.

**Frontmatter contract.** A workflow file is not a contract unless it has a parseable leading YAML frontmatter
block at all — `parse/2` (`workflows.ex:249-270`) returns `nil` (filtered out by `Enum.reject(&is_nil/1)`,
line 230) when `split_frontmatter/1` finds none or `parse_frontmatter/1` fails to parse it as YAML
(`workflows.ex:272-281`). No field is individually *required* to still count as a contract: `enabled` defaults
to `false` (`Map.get(fm, "enabled", false)`, line 260), `trigger`/`approval` default to `%{}`, `sources`
defaults to `[]` (lines 261-264). `risk_level` has no default (`nil` if absent). So the real gate is "has a
frontmatter block at all," not "has `id`/`inputs`/`target_path`" — those keys are convention (documented in
`priv/icm_template/AGENTS.md`), not schema-checked here.

**`get/2` containment.** `Valea.Workflows.get/2` (`workflows.ex:184-222`) resolves `mount_key` via
`Mounts.mount_by_key/2` (no enabled/healthy requirement — deliberate, see the moduledoc DECISION at
`workflows.ex:30-42`), then requires the resolved `relative_path` to land specifically inside *that mount's own*
`Workflows/` directory (`under_workflows_dir?/2`, `workflows.ex:242-247`) — a path like `Workflows/../Offers/x.md`
resolves inside the mount but outside `Workflows/` and is rejected as `:not_in_icm` (`workflows.ex:179-182`).

**Runner lifecycle.** `Valea.Workflows.Runner.run/4` (`backend/lib/valea/workflows/runner.ex:99-118`): fetch
workflow via `Workflows.get/2` → `ensure_enabled/1` (`runner.ex:436-437`, requires `enabled: true`) → resolve
input locator → `start_run/6`. Exact grants at session start (`runner.ex:612-629`): `write_paths: [staging_abs]`
(one exact file), `write_roots: [staging_dir/proposals]` (one exact subdirectory), `read_paths: [input_abs]` (one
exact file) — nothing else. The prompt (`runner.ex:680-695`) is a hardcoded English instruction string built from
three absolute paths; it names the workflow's `sources:` convention only informally ("Read only the pages its
Inputs and sources name").

**Proposal contract is hardcoded to one primary kind.** `Runner.valid_proposal?/1`
(`runner.ex:771-779`) requires `p["schema"] == "proposal/v1"` **and** `p["kind"] == "email_draft"` literally — a
workflow's primary output can *only* ever be an email draft today; any other kind fails validation outright
(`{:error, :invalid_pair}`-shaped, staging kept for inspection). The sidecar memory-update contract
(`Valea.Workflows.MemoryProposal`, `backend/lib/valea/workflows/memory_proposal.ex:76-83`) is a second, separate
schema (`memory_update/v1`: `target_path`, `base_sha256`, `reason`, `sources`) for `proposals/<name>.{md,json}`
pairs — validated independently, target containment done by `check_icm_target/2`
(`memory_proposal.ex:144-153`), scoped to the run's own owning ICM only.

**`run.json` sidecar.** Server-owned, written once at run start (`runner.ex:562-577,697-699`), carries
`run_id, workflow, workflow_hash, input, input_hash, risk_level, approval, created_at, icm_id, mount_key,
icm_root, session_id` — the agent's `write_roots` grant is scoped to `proposals/` only, *never* the staging dir
itself, so the sidecar is unwritable by the agent (`runner.ex:622-628`).

**Distill/digest special case.** `Valea.Workflows.Distill.digest/1` (`backend/lib/valea/workflows/distill.ex:26-55`)
compiles `queue/approved/*.json` + `queue/rejected/*.json` into an in-memory markdown digest — no real source
file exists. `Runner.run_generated/4` (`runner.ex:136-153`) exists *specifically* for this case: it writes the
generated bytes to a server-chosen staging path *before* the session starts (`materialize_input/3`,
`runner.ex:532-537`) so the agent still has a real file to point its prompt at. This is a workaround for the
"a workflow input must be an addressable ICM/workspace locator" assumption baked into `run/4`
(`Locator.resolve/2`-based) — a workflow whose input isn't a real file at all needs its own separate entry
point, not a variant of the normal one.

**FE consumption.**
- Workflows page (`frontend/src/routes/workflows/+page.svelte:1-40`): cards over the `list_workflows` RPC result;
  the "Distill recent decisions" button is matched to a specific card by exact `resolvedPath` string equality
  against the cockpit payload's `distillWorkflowPath` (comment at lines 22-30).
- Triage matching by **filename**: `frontend/src/lib/components/mail/triage-workflows.ts:29,49-58` —
  `TRIAGE_FILENAME = 'New Inquiry Triage.md'`; candidates are every enabled workflow whose
  `Path.basename(relativePath) === TRIAGE_FILENAME`, across every mount. Mount-agnostic (good), but still
  hard-pinned to one literal basename living directly under a top-level `Workflows/` dir.
- Cockpit `distillWorkflowPath`/`triageWorkflowPath`: `backend/lib/valea/cockpit.ex:24-57,78,175-178` — first
  match in `Workflows.list/0`'s own sort order, discovered purely by basename
  (`Valea.Workflows.triage_path/0`/`distill_path/0`, `workflows.ex:94-97,129-132`).

**What could NOT work today:**
- A workflow at `clients/kita/workflows/prep.md` (nested, lowercase `workflows/`) is invisible to
  `Workflows.list/1`'s glob (`workflows.ex:226-228` matches only `<mount>/Workflows/*.md`), so it can never be
  listed, triggered, matched by triage, or run via `run/4`/`get/2` (`under_workflows_dir?/2` would reject it even
  if addressed directly). It is just an ordinary page in the knowledge tree.
- A workflow that "wants to run a script" has no first-class contract for it: the Runner's prompt only ever
  instructs the LLM to read/write files (`runner.ex:680-695`); `PermissionPolicy.extract_paths/1`
  (`backend/lib/valea/agents/permission_policy.ex:408-414`) only looks at `file_path/path/notebook_path/filePath`
  keys in a tool call's `rawInput` — a `Bash` tool invocation carries none of those, so `paths == []` and it falls
  straight to `:ask` (`permission_policy.ex:119-120,276-277`) every single time, regardless of `session_kind`.
  There is no server-side notion of "this workflow's Process step is a deterministic script; capture its stdout as
  the proposal input" — everything is mediated through the same interactive ask-gate a chat session gets.

## 2. Knowledge-tree assumptions

**`tree_for/1` — md-only-ish, dot-dirs excluded, root `icm.yaml` hidden, other files listed as leaves.**
`Valea.ICM.build_tree/2` (`backend/lib/valea/icm.ex:240-252`):
- rejects any entry starting with `.` (line 243) — hidden dirs/files never appear in the tree at all;
- at the mount ROOT only, also rejects the literal `icm.yaml` file (line 248) — a nested folder's own file named
  `icm.yaml` is not special-cased and lists normally;
- `.md` → `:page` node with a `uri` (`icm.ex:269-275`);
- any other regular file → `:file` leaf with an `ext`, but *no* `uri` — "only `.md` pages are icm:// addressable/
  editable" (`icm.ex:283-291`);
- anything else (socket, fifo, dangling symlink) → dropped (`icm.ex:293-297`).

**Editor round-trip is markdown-only.** `Valea.ICM.page/1` (`icm.ex:56-87`) reads a file, splits frontmatter, and
converts the body through `Valea.Markdown.ProseMirror.from_markdown/2` — the only content type the editor can
open/save at all. A non-`.md` file is a passive tree leaf, never opened by this path.

**Search is recursive `**/*.md`, no depth limit — the one place Valea does *not* impose the top-level-only rule.**
`Valea.ICM.Search.scan_mount/3` (`backend/lib/valea/icm/search.ex:92-103`) globs `<root>/**/*.md` — any depth.
Worth noting explicitly as a contrast: the knowledge content itself is not depth-restricted; it's specifically
Workflows/Templates/References (below) that are.

**References (workflow-reference tracking for rename) — top-level `Workflows/*.md` only, non-recursive.**
`Valea.ICM.References.workflows_dir/1` + `workflow_files/1` (`backend/lib/valea/icm/references.ex:262-269`) scan
only `<mount.root>/Workflows/*.md` — the exact same single-segment glob as the registry. A workflow living at any
other depth is invisible to rename-driven reference rewriting (`Valea.ICM.rename/3` calls this, `icm.ex:589-621`)
— renaming a page never updates a nested workflow's `sources:` frontmatter that pointed at the old path.

**Backlinks — recursive, but scope-limited to declared-related ICMs, not depth-limited.**
`Valea.ICM.Backlinks.backlinks/2` (`backend/lib/valea/icm/backlinks.ex:50-75`) scans `**/*.md` recursively across
`Mounts.scoped_roots/2` (primary + directly-declared-related mounts only, per §3 below) — any depth within scope,
but a link living in an unrelated mounted ICM is never discovered (moduledoc `backlinks.ex:12-19`, "by design, not
a gap").

**Templates/ — top-level-only, and enforced purely in the frontend, not the backend.**
`frontend/src/lib/components/knowledge/template-options.ts:30-40`: the "Start from template" picker finds
`group.tree.find(n => n.type === 'folder' && n.name === 'Templates')` among a mount's **top-level** tree nodes
only (line 34) — a nested `clients/kita/Templates/` folder is invisible to this UI. The backend RPC,
`Valea.ICM.create_page_from_template/5` (`icm.ex:460-514`), has *no* such restriction itself — it only requires
`template_mount_key == mount_key` (same-mount, line 441-442, `read_template/5` at 499-507) and ordinary
containment; the top-level convention exists only in the FE discovery layer.

**Decisions/ convention — zero code enforcement anywhere.** A repo-wide grep for `"Decisions"` across
`backend/lib/valea` and `frontend/src/lib` returns no hits outside the seed template
(`backend/priv/icm_template/Decisions/2026.md`, `AGENTS.md`'s prose). It is pure documentation/seed convention;
nothing parses, lists, or specially treats a `Decisions/` folder.

**Assets/ image convention — hardcoded, flat, mount-root-only.** `ValeaWeb.FilesController.write_and_respond/4`
(`backend/lib/valea_web/controllers/files_controller.ex:113-131`) always computes
`dest_rel = Path.join("Assets", filename)` (line 115) — every pasted/uploaded image, regardless of how deep the
page being edited lives in the tree, lands in one flat `Assets/` folder at the mount's own root, disambiguated
only by a `<page-slug>-<hash8>.<ext>` filename (line 114). There is no per-folder or mirrored-nesting `Assets/`.
The moduledoc (`files_controller.ex:25-48`) explicitly documents this as a reviewed, deliberate exception to
"Valea never writes into a user-owned ICM" — user-initiated content, not Valea metadata — but the flat-root
placement itself is not configurable.

## 3. ICM identity & routing contracts

**icm.yaml format-2 requirement.** `Valea.Mounts.Manifest.load/1` (`backend/lib/valea/mounts/manifest.ex:44-63`)
requires `id` to parse as a UUID (`fetch_id/1`, lines 113-124) and `name` to be a non-blank string (`fetch_name/1`,
lines 126-138); `format` defaults to `2` but a *present* value — including legacy `format: 1` — is preserved
as-is and still loads (moduledoc, lines 14-18). `Valea.Mounts.mount/2`
(`backend/lib/valea/mounts.ex:294-308`) delegates to `validate_mountable/2` (`mounts.ex:426-443`), which requires
`Manifest.load/1` to succeed at all — a folder with no `icm.yaml` returns `{:error, :no_manifest}` from `mount/2`
and **cannot be mounted** via the `mount_icm` RPC.

**A folder without icm.yaml, if hand-added to `config/workspace.yaml`'s `icms:` map directly:** `list/1`'s
`build_resolved_icm_mount/4` (`mounts.ex:665-690`, specifically the `"icm.yaml is missing"` branch around line
680) degrades it — it stays *listed* (for the UI/doctor to show something is wrong) but is unconditionally
excluded from `enabled/0,1` (`mounts.ex:117-129`) regardless of its config `enabled` flag.

**`inspect_icm` reason.** `Valea.Api.Icms.load_and_validate_manifest/1`
(`backend/lib/valea/api/icms.ex:326-345`) returns `%{"ok" => false, "reason" => "no icm.yaml found in that
folder"}` (lines 339-340, 347-349) for a folder with no manifest — this is the onboarding preview primitive, run
before any workspace exists (moduledoc, lines 55-89); it *additionally* rejects a loadable-but-`format: 1`
manifest (lines 336-337, "expected format 2") even though `Manifest.load/1` itself would accept it — a stricter
gate than actual mounting enforces.

**Root `CONTEXT.md` `related_icms` contract.** `Valea.Mounts.Context.resolve/2`
(`backend/lib/valea/mounts/context.ex:64-76`) reads `<primary_mount.root>/CONTEXT.md`'s frontmatter
(`read_related_icms/1`, lines 80-85 — **root only**, no nested `CONTEXT.md` is ever consulted for this purpose).
`parse_related_icms/1` (lines 87-99) requires `format` to be exactly `1` (default) — `1 <- Map.get(doc, "format",
1)` (line 93) — any other value silently yields `[]` (soft-fail, not an error). Each declared entry
(`id`/`name`/`entrypoint`) resolves `id` against `Mounts.mount_by_id/2` (healthy + enabled required,
`context.ex:132-138`) and `entrypoint` (default `"CONTEXT.md"`) against the *related* ICM's own root via
`Paths.resolve_real/2` — an escaping entrypoint is a hard reject (`:entrypoint_escapes`, lines 154-168). Direct-
only, cycle-safe by construction — a related ICM's own `CONTEXT.md` is never read (moduledoc, lines 27-31).

**Documentation/implementation mismatch worth flagging separately:** the seed template's own
`priv/icm_template/CONTEXT.md` (commented block near the end) describes a *different* convention — a **page-level**
frontmatter key `related_icms: [legal]` on any page anywhere — which `Valea.Mounts.Context` does not implement at
all (it only ever reads root `CONTEXT.md`'s `related_icms:` list of `{id, name, entrypoint}` maps). The shipped
template teaches a contract the code doesn't have.

**AGENTS.md/CLAUDE.md role — Valea does not read their content; only the harness does.** A repo-wide grep
confirms no module in `backend/lib/valea` parses AGENTS.md/CLAUDE.md bodies. Valea's own role is limited to: (a)
seeding them from `priv/icm_template` at `create/3` time (`seed_template!/2`, `mounts.ex:523-535`), (b) treating
their exact top-level basenames as `"high"` `RiskTier` behavior files (`backend/lib/valea/agents/risk_tier.ex:26`),
and (c) always-allowing their relative read via `PermissionPolicy`'s `@root_files` bypass
(`backend/lib/valea/agents/permission_policy.ex:81,222-224`). The actual `@AGENTS.md`/`@CONTEXT.md` import
mechanism referenced by `priv/icm_template/CLAUDE.md` (`"@AGENTS.md\n@CONTEXT.md"`) is pure Claude-Code-harness
convention that Valea's own code has zero awareness of.

**Template seeding content.** `backend/priv/icm_template/` (`AGENTS.md`, `CLAUDE.md`, `CONTEXT.md`, `icm.yaml`,
`Decisions/2026.md`, `Templates/Client.md`, `Templates/Decision.md`, `Workflows/Distill Decisions.md`) — every
freshly-created ICM (`Valea.Mounts.create/3`) is pre-committed to exactly the top-level `Workflows/`/`Templates/`/
`Decisions/` shape whose rigidity is under review here, from the moment it's minted.

## 4. Agent session contract

**`SessionScope.resolve/1` pipeline** (`backend/lib/valea/agents/session_scope.ex:56-108`): generation check
(fails first on stale generation) → workspace lookup → `mount_by_key` (enabled + non-degraded required,
`session_scope.ex:67-72`) → `Context.resolve/2` for DIRECT related ICMs only (issues attached, not fatal) →
`cwd` is **always** the primary ICM's root, never the workspace root, never caller-supplied (line 93, moduledoc
line 25-26) → harness `launch/2` materializes `context.md` and computes managed-settings posture.

**Injected `context.md` content** (`Valea.Agents.SessionSettings.context/1`,
`backend/lib/valea/agents/session_settings.ex:106-131`): primary ICM identity + root, a list of related ICMs
(mount_key/root/entrypoint), and — **only when `scope.kind == "workflow"`** — the entire `@workflow_contract`
block appended verbatim (lines 23-76, 126-131). This hardcoded English + JSON-schema text (the `proposal/v1` and
`memory_update/v1` shapes) is the *only* place a workflow session is taught those schemas — it lives in Elixir
source, not anything an ICM can declare or override for itself.

**Posture rules — the workspace deny-list never reaches inside an ICM.**
`SessionSettings.content/1` (`session_settings.ex:78-104`): `deny` covers only
`<workspace_root>/{logs,config,secrets,runtime,.git}` + `<workspace_root>/app.sqlite*` + `WebFetch`/`WebSearch`
(lines 90-95). `PermissionPolicy.split_protected?/2` (`permission_policy.ex:181-186`) is likewise scoped
*exclusively* to `workspace_root` (`@protected_dirs`, line 79). **Confirmed directly: an ICM-internal `secrets/`
directory or `.env` file gets no deny-list treatment at all** — `Valea.Mounts.Doctor.secrets_hygiene_check/1`
(`backend/lib/valea/mounts/doctor.ex:345-361`) only *warns* (status `"failed"` in display terms, but nothing is
ever blocked — moduledoc lines 61-70: "Valea's workspace deny-list does not reach into an external folder it
doesn't own").

**RiskTier path tiering — exact-match top-level allowlist, not depth-aware.**
`Valea.Agents.RiskTier.classify/1` (`backend/lib/valea/agents/risk_tier.ex:26,35-41`): `"high"` only for a path
that is *exactly* `"AGENTS.md"`, `"CLAUDE.md"`, or `"icm.yaml"` (`@behavior_files`, exact string match, not a
prefix) or that starts with the literal prefix `"Workflows/"`; `"medium"` for **everything else** inside the same
ICM. A workflow at `clients/kita/workflows/prep.md`, or an instruction-bearing file two levels deep, is
indistinguishable in risk tier from an ordinary client note.

**PermissionPolicy split-contract highlights** (`permission_policy.ex:93-135`): deny → allow → ask, deny always
wins. Read allowed only if every candidate resolves inside some `read_root`, or is a bare `AGENTS.md`/`CLAUDE.md`
relative to `cwd` (`split_root_file?/2`, line 222-224). Write allowed **only** when `session_kind == "workflow"`
**and** every candidate is inside `write_paths`/`write_roots` — i.e. only the run's own staging area; a workflow
session can *never* write directly into ICM content — every content change must flow through the memory_update
proposal + human-approval pipeline. A legacy pre-split contract (`decide_legacy/2`, lines 254-388) still lives
alongside this, pending deletion once every caller migrates.

## 5. Mail/calendar as-built

**Mail subsystem shape** (`backend/lib/valea/mail/*`): `Engine` (GenServer, per-workspace sync/poll/credential
state, `engine.ex`), `ImapClient`/`Transport`/`imap/wire.ex` (raw IMAP wire protocol), `SyncPass` (one poll cycle),
`Normalizer` (header parsing), `MessageFile` (message ⇄ markdown mirror), `MailboxOps` (post-approval draft-append
+ archive-move), `Store` (+ `store/{sync_state,uid_outcome,message_index,inbox_header}.ex`, SQLite cache),
`Settings` (`config/mail.yaml` v3), `Doctor`, `DraftMime`, `Redact`.

**IMAP, not maildir.** `Valea.Mail.Settings` (`backend/lib/valea/mail/settings.ex:1-30`) is an IMAP account
config (host/port/username, TLS mandatory, no toggle). `scripts/dovecot/dovecot.conf` is a local dev/test IMAP
*server* Valea talks to over the wire — not a filesystem maildir Valea reads directly. There is no maildir
building block anywhere in this codebase today.

**Where messages land on disk.** `Valea.Mail.MessageFile.render/2`
(`backend/lib/valea/mail/message_file.ex:118-142`) synthesizes a **Valea-invented** YAML-frontmatter markdown
file per message at `sources/mail/messages/<msg_id>.md` (frontmatter fields: `id, message_id, from, to, subject,
date, uid, in_reply_to, references, reply_to, status, source, source_ref, attachments`, injection-hardened via
`yaml_string/1`, lines 184-206) — this is **not** a raw `.eml`/maildir file, it is a bespoke mirror format. Drafts
land at `sources/mail/drafts/<run_id>.md` (`mailbox_ops.ex:173-174`); attachments under
`sources/mail/attachments/` (workspace template).

**SQLite index.** `Valea.Mail.Store` (`backend/lib/valea/mail/store.ex:1-15`) — four Ash resources
(`mail_sync_state`, `mail_uid_outcomes`, `mail_messages`, `mail_inbox_headers`), explicitly "pure cache: rebuildable
from `sources/mail/` (+ an IMAP resync)."

**MessageView "Run triage" action.** `frontend/src/lib/components/mail/triage-workflows.ts:49-58` — candidates
are every *enabled* workflow across every mount whose `relativePath` basename equals the literal
`"New Inquiry Triage.md"` — filename-matched, not capability- or content-matched (mirrors §1's registry rigidity).

**How a message becomes a workflow input.** `Runner.run/4` takes an `input_locator` — for mail, always
`%{"kind" => "workspace", "path" => "sources/mail/messages/<id>.md"}`, resolved via
`Valea.Icm.Locator.resolve/2` (`backend/lib/valea/icm/locator.ex:116-119`) then read once
(`resolve_input/2`, `runner.ex:493-500`). The agent session's grant is `read_paths: [input_abs]` **only**
(`session_scope.ex:51,94`) — the agent cannot browse `sources/mail/` freely; it sees exactly the one file it was
handed, nothing else in the mailbox.

**Calendar as-built: a stub.** `config/calendar.yaml` (`backend/priv/workspace_template/config/calendar.yaml`)
declares `account`/`caldav`/`ics_fallback`/`event_types`, and `sources/calendar/events/.gitkeep` exists as a
placeholder — but a repo-wide grep for calendar-domain code in `backend/lib/valea` returns **zero hits** (the two
matches found, `Calendar.strftime` in `draft_mime.ex` and a watcher comment, are unrelated). No Engine, no sync,
no store, no domain module exists. The config vocabulary is pre-committed before any implementation.

**The "sources" concept.** `backend/priv/workspace_template/sources/{calendar/events, files, mail/{attachments,
drafts,messages,inbox.md}}` — a fixed, backend-defined directory taxonomy created at workspace-scaffold time, not
something an arbitrary ICM or agent grows organically. `Valea.ICM.Watcher` watches the whole `sources/` tree but
"a change under `sources/` produces no event at all" beyond mail/queue-specific wiring
(`backend/lib/valea/icm/watcher.ex:1-24`).

**Keychain/credential handling shape.** `frontend/src/lib/keychain.ts:1-69` — the sole module allowed to touch
Tauri IPC for this purpose, wrapping `desktop/src-tauri/src/keychain.rs`'s `keyring`-backed commands; service =
app bundle id, account = `workspace_id:username` (stable across folder moves). Browser/dev fallback is the
`VALEA_MAIL_PASSWORD` env var, read **server-side only** by `Valea.Mail.Engine.env_credential/0`
(`engine.ex:432-437`) — the frontend module plays no part in that path. The credential is held in GenServer state
as a **zero-arity closure** (`fn -> secret end`, not the raw string) specifically so `:sys.get_state/1`/crash
dumps never leak it (`engine.ex:23-32`).

## 6. Queue/approval coupling

**Proposal kinds: exactly two, hardcoded in at least three independent places.**
1. `Runner.valid_proposal?/1` (`runner.ex:771-779`) — the *primary* `proposal.json` output of any workflow run
   must have `p["kind"] == "email_draft"` literally. No other primary kind can ever validate today.
2. `MemoryProposal.valid_manifest?/1` (`memory_proposal.ex:76-83`) — the sidecar `proposals/<name>.json` pairs use
   a wholly separate `memory_update/v1` schema.
3. `Queue.valid_payload?/1` → `valid_action_for_kind?/2` (`backend/lib/valea/queue.ex:685-718`) — `"memory_update"`
   gets its own action-shape clause (lines 705-717); **every other kind falls through to `valid_action?/1`**
   (line 718), which only recognizes `{"type" => "create_email_draft", "to", "subject", "body_markdown"}`
   (lines 726-735). A payload with any third `kind` string simply fails validation (`valid_action?/1`'s catch-all,
   line 735).
4. `Queue.approve/2`'s kind dispatch (`queue.ex:189-192`) is written as `case item["payload"]["kind"] do
   "memory_update" -> execute_memory(...); _ -> execute_email(...) end` — a **2-way**, not N-way, dispatch: *any*
   kind that isn't literally `"memory_update"` executes the email-draft flow. A hypothetical third kind added to
   the validator (#3) without also touching this line would be accepted by validation and then silently routed
   into `execute_email/3`, which composes an RFC822 draft from fields that kind might not even have.

**How hard to add a new kind:** touches all four call sites above, plus the frontend's queue-item
rendering/diff-preview (not itself surveyed here, but structurally coupled to the same two shapes via
`memory_display_fields/2`, `queue.ex:600-...`, and the FE's own type definitions).

**Apply executor's assumptions.** `Queue.apply_page_content/2` (`queue.ex:262-306`, the `memory_update` executor)
assumes the target is *always* an ICM locator (`valid_target_locator?/1` only accepts `"kind" => "icm"`, line
720-722 — a workspace-locator memory-update target is impossible by construction) carrying markdown page content,
guarded by a `base_sha256` hash check (`check_base/2`, lines 296-306: `nil` means create-only, mismatch means
conflict bounced back to `pending/`). There is no notion of any other target shape — "run a script," "append to a
non-markdown file," "call an external API" have no executor and no validation path.

## 7. Harness seam

**`Valea.Harness` behaviour** (`backend/lib/valea/harness.ex:17-21`) — three callbacks: `definition/0`,
`acp_command/1`, `launch/2`. Genuinely minimal on paper (moduledoc: "adding an agent is ~10 lines").

**Exactly one implementation exists, and it is not generic in practice.**
`Valea.Harnesses.ClaudeCode` (`backend/lib/valea/harnesses/claude_code.ex`) is the *only* module implementing the
behaviour. It spawns the `@agentclientprotocol/claude-agent-acp` adapter specifically (moduledoc, line 3) and
conveys the permission posture via a **Claude-Code-specific ACP meta channel**
(`_meta.claudeCode.options.managedSettings`, comment at lines 27-33) that only this adapter understands — nothing
in the `Harness` behaviour abstracts that channel; a second harness would need its own equivalent mechanism
invented from scratch. `Valea.Agents.Env.allowlist/0` (`backend/lib/valea/agents/env.ex:8`) bakes
`ANTHROPIC_API_KEY` and `CLAUDE_CODE_OAUTH_TOKEN` directly into the "minimal" subprocess env allowlist — provider-
specific credentials treated as first-class, generic plumbing. No second harness implementation exists anywhere
in the tree to prove the seam actually generalizes to a different agent CLI/provider.

## Top 10 rigidity points, ranked by user impact

1. **Workflows are discoverable only at `<mount>/Workflows/*.md` — one fixed top-level folder, capital W,
   non-recursive.** (`workflows.ex:226-231`, mirrored in `references.ex:262-269` for rename-rewrite, and in the
   FE's filename-matched triage discovery.) This is the single biggest blocker to "workflows at any depth,
   lowercase, freeform" — the reference-ICM study found real workflows living 2-4 levels deep, wherever the
   domain content lives. Everything downstream (registry, Runner, RiskTier's `"Workflows/"` prefix, References,
   FE triage/distill matching) assumes this one location.

2. **Mail is a bespoke synthetic markdown-mirror + SQLite cache over IMAP, not a filesystem primitive the agent
   can browse.** (`message_file.ex:118-142`, `store.ex:1-15`.) Directly contradicts the "email as a filesystem
   building block (maildir)" hypothesis — there is no maildir anywhere in the stack, and the agent never sees the
   mailbox itself, only one exact-granted file per run (`session_scope.ex:51,94`).

3. **The proposal/queue vocabulary is hardcoded to exactly two kinds, enforced independently in ≥4 places**
   (`runner.ex:771-779`, `memory_proposal.ex:76-83`, `queue.ex:685-718`, `queue.ex:189-192`) — and the approval
   dispatch is a *2-way* `"memory_update" | _` switch, so an incompletely-added third kind would silently execute
   as an email draft rather than failing loudly.

4. **No script/process-execution contract in the workflow model at all.** A workflow that "wants to run a
   script" has nothing to declare that with; every `Bash` tool call hits `:ask` unconditionally because
   `PermissionPolicy.extract_paths/1` only recognizes file-path keys (`permission_policy.ex:408-414`), never a
   command/script identity.

5. **RiskTier's behavior-file detection is an exact-match, top-level-only allowlist.**
   (`risk_tier.ex:26,35-41`.) A nested or renamed instruction-bearing file is silently "medium" risk — identical
   to an ordinary client note — which is both a rigidity gap (assumes flat structure) and a quiet trust/security
   gap.

6. **Calendar is 100% unimplemented, yet its config vocabulary is already pre-committed.**
   (`calendar.yaml` template, `sources/calendar/events/` placeholder, zero domain code.) Low cost to redesign
   today since nothing exists to migrate — but the shape of what *would* be built is already implied by the
   scaffold before any implementation decision is made.

7. **Templates/ discovery is frontend-only and top-level-tree-node-only**
   (`template-options.ts:30-40`) even though the backend RPC itself (`icm.ex:460-514`) has no such restriction —
   an easy, low-risk fix that's purely a UI gap, not an architectural one.

8. **Reference-rewrite-on-rename only scans top-level `Workflows/*.md`**, non-recursive
   (`references.ex:262-269`) — a nested workflow's `sources:` frontmatter silently goes stale on a page rename,
   with no error or warning surfaced anywhere.

9. **Uploaded images always flatten into one mount-root `Assets/` folder** regardless of how deep the page being
   edited lives (`files_controller.ex:113-131`) — an ICM with real subject-matter nesting gets all images dumped
   into a single flat namespace, disambiguated only by a generated filename.

10. **The harness seam has exactly one implementation, and it leaks Claude-Code-specific mechanics into what's
    billed as the generic launch path** (`claude_code.ex:27-46`, `env.ex:8`) — its genericity is asserted in the
    moduledoc but unproven by any second implementation.

Honorable mention (documentation/implementation drift, not runtime rigidity): the shipped `CONTEXT.md` template
comment (`priv/icm_template/CONTEXT.md`) teaches a page-level `related_icms:` convention that
`Valea.Mounts.Context` does not implement — only a root-level `CONTEXT.md` with `format: 1` and a list-of-maps
`related_icms:` is ever read (`context.ex:64-99`).
