# Release pipeline & auto-update — design

Date: 2026-07-19
Status: approved design, implemented same day
Scope: infra (CI + desktop shell + sidebar UI), no product-domain changes

## Context & goal

Valea ships as a Tauri v2 desktop app whose entire backend is a Burrito
sidecar binary (`valea-server-<triple>`), built so far only by hand and only
for Apple-silicon macOS (`just package-backend` → `just desktop-bundle`).
There is no CI, no published release, and no way for an installed app to
learn that a newer version exists.

Goal: a tag-driven GitHub Actions pipeline that builds signed, updatable
desktop bundles per platform and publishes them as a GitHub Release, plus an
in-app auto-update flow — a quiet notice at the bottom of the left sidebar
when a new version is out, background download, one click to install and
relaunch.

## Approaches considered

1. **Custom checker** (poll GitHub API, link to the download page). Trivial
   to build, but no signature story and no install/relaunch — the user
   re-downloads by hand. Rejected: the ask is explicitly auto-download +
   relaunch.
2. **Tauri updater plugin + GitHub Releases + `tauri-action`** ← chosen.
   First-class in the stack we already ship: minisign-signed artifacts,
   `latest.json` manifest generated by CI, `check()/download()/install()`
   JS API, relaunch via the process plugin. The repo is public, so the
   `releases/latest/download/latest.json` endpoint needs no auth.
3. **Hosted update server** (CrabNebula Cloud or self-hosted). Adds an
   account/service dependency for nothing we need at this scale. Rejected.

## Platform matrix

| Target | Runner | Burrito target | Bundles | Auto-update |
|---|---|---|---|---|
| macOS Apple silicon (`aarch64-apple-darwin`) | `macos-latest` | `macos_arm` | `.app` → `.dmg` | `.app.tar.gz` + sig |
| Linux x86_64 (`x86_64-unknown-linux-gnu`) | `ubuntu-22.04` | `linux_x64` | AppImage, `.deb`, `.rpm` | AppImage + sig |
| Windows x86_64 | — | staged, **disabled** | — | — |

Decisions and their reasons:

- **No Intel macOS.** GitHub retired the last Intel runners (`macos-13`);
  cross-compiling is out because the sidecar embeds natively-compiled NIFs
  (exqlite's C sqlite3, erlexec's C++ port program) — every sidecar must be
  built on hardware matching its target. Apple silicon covers every Mac
  sold since 2020.
- **Native builds only, one Burrito target per runner.** `BURRITO_TARGET`
  pins the single target matching the host; we never ask zig to
  cross-compile a release whose NIFs were compiled for the build host.
- **Linux builds on `ubuntu-22.04`**, the oldest supported image, so the
  AppImage links against the oldest glibc we can get for free.
- **Windows is genuinely blocked in the app, not the pipeline**, so its
  Burrito target stays commented out rather than shipping a boot-crashing
  binary. Blockers (tracked in docs/RELEASING.md): `erlexec` is Unix-only
  and starts with the OTP application at boot; the Spec E maildir stores
  flags in `:2,`-style filenames, and `:` is illegal on NTFS; the 3-layer
  prose seed writes a `CLAUDE.md` symlink. Windows support is a product
  effort with its own spec, not a CI flag.

## Release flow

- Push a tag `vX.Y.Z` (must equal `version` in
  `desktop/src-tauri/tauri.conf.json`; the workflow asserts this and fails
  fast on drift). `workflow_dispatch` runs build-only — no release, bundles
  land as workflow artifacts — for testing the pipeline without tagging.
- Per-platform jobs assemble the SPA → bake into Phoenix `priv/static` →
  Burrito sidecar (`just package-backend`, now host-aware) →
  `tauri-action` builds bundles + updater artifacts and uploads everything
  to one **draft** release, merging each platform into `latest.json`.
- Publishing the draft is the manual go-live act: `releases/latest/…` only
  serves published releases, so nothing reaches updaters until Daniel
  publishes. That's the rollout safety valve.
- Toolchain per job: `.tool-versions` is the single version truth —
  `setup-beam` reads it on Linux; macOS (unsupported by `setup-beam`)
  installs via asdf with the whole `~/.asdf` cached, compiling OTP once per
  version bump. Bun via `setup-bun`, Rust stable + `rust-cache`, zig 0.15.2
  auto-provisioned by the existing `build-release.sh` (cached).

## Update artifacts & signing

- `bundle.createUpdaterArtifacts: true`; every updater artifact is minisign
  signed. The public key lives in `tauri.conf.json`; the private key exists
  only in `~/.tauri/valea_updater.key` on Daniel's machine and in the
  `TAURI_SIGNING_PRIVATE_KEY` GitHub secret (password secret alongside).
  The key never enters the repo. Losing it means shipping one manual
  "reinstall by hand" release with a new key — annoying, not fatal.
- macOS bundles are ad-hoc signed for now. The updater path doesn't care
  (minisign, no quarantine xattr on its downloads), but **first** installs
  on other people's Macs hit Gatekeeper; Apple Developer ID + notarization
  is a documented follow-up — the workflow already passes the `APPLE_*`
  env slots through to `tauri-action`, so adding the secrets is the whole
  job.

## In-app update flow

- **Endpoint**: `https://github.com/wir-drei-digital/valea/releases/latest/download/latest.json`.
- **Boundary module** `frontend/src/lib/updater.ts` — the second (and last)
  module allowed to touch Tauri IPC, alongside `keychain.ts`, same
  contract: browser/vitest are documented no-ops, nothing ever throws.
  Wraps `@tauri-apps/plugin-updater` + `@tauri-apps/plugin-process`.
- **Store** `frontend/src/lib/stores/updates.svelte.ts` — phases
  `idle → checking → (none | downloading → ready | error)`, then
  `installing` on user action. Checks ~90 s after boot (stay out of the
  workspace bootstrap's way), re-checks every 6 h; an available update
  auto-downloads immediately (the "auto download" half of the ask).
  Wired once from the root `+layout.svelte`, gated to the packaged desktop
  app (`inDesktop() && PROD`) — browser dev and `tauri dev` never see it.
- **UI** `UpdateNotice.svelte` at the bottom of the sidebar footer, above
  the workspace switcher. Amber "suggest" card per the design system
  (updates are a suggestion, installing is the act): downloading shows a
  thin progress track; ready shows "Restart to update" (an act-green
  button) → `install()` + `relaunch()`; failure shows one quiet line with
  "Try again". No modal, no badge elsewhere, hidden entirely when idle.
- **Sidecar lifecycle on relaunch**: `RunEvent::Exit` already kills the
  sidecar; `relaunch()` goes through normal exit, so the new app boots a
  fresh sidecar — no port-4817 collision with a zombie.
- **IPC surface**: new `updates` capability grants `updater:default` +
  `process:allow-restart` to the main window including the remote loopback
  origins (`http://localhost:4817` / `:4273`), mirroring `mail-keychain`.

## Versioning

`desktop/src-tauri/tauri.conf.json` `version` is the app version the
updater compares against `latest.json`. Backend `mix.exs` / package.json
versions are internal and stay untouched. Cutting a release = bump
tauri.conf.json version, commit, tag `v<same>`, push tag, publish the
draft. CI enforces tag ↔ version equality.

## Testing

- `updater.test.ts` mirrors `keychain.test.ts`: mock the two Tauri plugin
  packages, prove browser no-ops never call the bridge and desktop paths
  invoke with exact args and never throw.
- `updates.test.ts` drives the store with a mocked `$lib/updater`:
  available → auto-download → ready; up-to-date → hidden; check/download
  failures → error phase with retry; install path calls install then
  relaunch.
- Pipeline verification is a `workflow_dispatch` dry run (build-only), then
  a real `v0.1.x` tag producing a draft release with both platforms in
  `latest.json`.
